# Calango Contexts — Phoenix-style domain grouping

> Status: Approved for implementation
> Branch: feature/contexts-pattern

---

## Overview

Calango adopts the **contexts** pattern from Phoenix Framework to replace the flat resource structure generated by `calango generate resource`. Instead of layering across the entire `app/` directory (all models together, all services together), resources are grouped by **domain context** — similar to DDD bounded contexts.

This change affects only `calango-cli` (templates and the `generate` command). `calango-core` (BaseRepository, BaseService) is unchanged.

---

## Design Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Contexts are mandatory | Yes — no flat fallback | Forces domain thinking from the first resource |
| CLI syntax | `calango generate resource Accounts.User` | Dot notation — concise, familiar from OOP |
| Internal structure | Subfolders by type (`models/`, `schemas/`, `repositories/`, `services/`) | Familiar to devs coming from the old flat structure |
| Router location | `app/routers/` (outside context) | Same as Phoenix: web layer separate from domain |
| Public API | `app/contexts/<domain>/__init__.py` | Auto-created and auto-updated on each resource addition |

## Resource Generation

### Command

```bash
calango generate resource Accounts.User
```

Parsing:
- `Accounts` → `context_name="Accounts"`, `context_snake="accounts"`
- `User` → `resource_name="User"`, `resource_snake="user"`, `resource_plural="users"`
- Without dot notation → exit 1 with hint: `"Use the format Context.Resource: calango generate resource Shop.Order"`

### Generated Files (10 total)

```
app/contexts/accounts/
├── __init__.py                          ← created (new context) or re-rendered (existing context)
├── models/user.py
├── schemas/user.py
├── repositories/user.py
└── services/user.py

app/routers/user.py                      ← imports from app.contexts.accounts

tests/unit/accounts/test_user_service.py
tests/integration/accounts/test_user_router.py
tests/factories/user_factory.py
```

## Context Public API (`__init__.py`)

```python
# Accounts context — public API
# Do not import sub-modules directly outside this package.

from app.contexts.accounts.schemas.user import UserInput, UserOutput, UserUpdate
from app.contexts.accounts.repositories.user import UserRepository
from app.contexts.accounts.services.user import UserService

__all__ = ["UserInput", "UserOutput", "UserUpdate", "UserRepository", "UserService"]
```

### Multi-resource contexts

When `calango generate resource Accounts.Session` is run after `Accounts.User`:
1. Existing `__init__.py` is read; registered resources are extracted via regex
2. New resource is added to the list (deduped, sorted alphabetically)
3. Entire `__init__.py` is re-rendered from `context_init.py.jinja` with the full list

```python
# Accounts context — public API
# Do not import sub-modules directly outside this package.

from app.contexts.accounts.schemas.session import SessionInput, SessionOutput, SessionUpdate
from app.contexts.accounts.repositories.session import SessionRepository
from app.contexts.accounts.services.session import SessionService

from app.contexts.accounts.schemas.user import UserInput, UserOutput, UserUpdate
from app.contexts.accounts.repositories.user import UserRepository
from app.contexts.accounts.services.user import UserService

__all__ = [
    "SessionInput", "SessionOutput", "SessionUpdate", "SessionRepository", "SessionService",
    "UserInput", "UserOutput", "UserUpdate", "UserRepository", "UserService",
]
```

## Import Convention

```python
# ✅ Router imports from context public API
from app.contexts.accounts import UserService, UserRepository, UserInput, UserOutput

# ❌ Direct cross-context internal import — FORBIDDEN
from app.contexts.catalog.models.product import Product
```

## Template Changes

| Template | Change |
|---|---|
| `model.py.jinja` | No change (no app imports) |
| `schemas.py.jinja` | No change (no app imports) |
| `repository.py.jinja` | Import model from `app.contexts.{ctx}.models.{res}` |
| `service.py.jinja` | Import from `app.contexts.{ctx}.repositories.{res}` and `app.contexts.{ctx}.schemas.{res}` |
| `router.py.jinja` | Import everything from `app.contexts.{ctx}` (public API) |
| `test_service.py.jinja` | Import service from `app.contexts.{ctx}.services.{res}` |
| `test_router.py.jinja` | No change (uses client fixture only) |
| `factory.py.jinja` | Import model from `app.contexts.{ctx}.models.{res}` |
| `context_init.py.jinja` | NEW: renders context `__init__.py` for a list of resources |
